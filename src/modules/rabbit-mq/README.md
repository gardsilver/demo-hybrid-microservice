# RabbitMq Module

## Описание

Реализован базовый функционал для создания **RabbitMq**-клиента или **RabbitMq**-сервера.

- **RabbitMq Common** основной функционал для работы с **RabbitMq** **Request/Response**.
- **RabbitMq Server** модуль для настройки **RabbitMq**-сервера: **Middleware**, **Guards**, **Interceptors**, **Pipes** и т.д.
- **RabbitMq Client** модуль для настройки и создания **RabbitMq**-клиента.

### Особенности

Специфика реализации  **RabbitMq**-сервера, а именно механики **Request/Response** [NestJS RabbitMq](https://docs.nestjs.com/microservices/rabbitmq) имеет ряд особенностей, которые разрешены в данной реализации. А именно:

- Необходимо подключать каждый консъмер **RabbitMq**-сервера по отдельности.

```typescript
app.connectMicroservice<MicroserviceOptions>({
    transport: Transport.RMQ,
    options: {
        urls: ['broker1', 'broker2'],
        queue: 'queue1',
        ....
    }
});
app.connectMicroservice<MicroserviceOptions>({
    transport: Transport.RMQ,
    options: {
        urls: ['broker1', 'broker2'],
        queue: 'queue2',
        ....
    }
});
```

И указанные настройки в `connectMicroservice` должны быть согласованными с декоратором **EventPattern**, что в целом усложняет общую конфигурацию  **RabbitMq**-сервера.

В данной реализации требуется подключить **RabbitMq**-сервер один раз с указанием базовых настроек, которые будут применены по умолчанию к каждому подключенному косъюмеру через декоратор `EventRabbitMqMessage` (данный декоратор позволят более детально настроить косъюмер).

- Процесс **Request/Response** **NestJS RabbitMq** подразумевает, что полученный ответ на отправленный запрос будет вычитан консъюмером того же **Pod**-а, что требует дополнительной согласованности всех настроек клиента и сервера и увеличивает общую сложность интеграций через **RabbitMq**.

В данном решении реализованы независимые друг от друга способы отправки сообщений и обработки полученных сообщений. Соответственно для полноценной реализации подходов **Request/Response** может возникнуть необходимость дополнительно использовать общий ресурс ваших **Pod**-ов (базу данных например) для фиксации информации об отправленном сообщении, что бы консъюмер понимал, что полученное сообщение является ответом на запрос от другого **Pod**-а. Такой подход увеличивает надежность обмена сообщениями через **RabbitMq** и в случаях сбоев позволит автоматически восстанавливать работу приложения.

- В целом высокая сложность настройки **NestJS RabbitMq** из-за непрозрачности типизации опций для разных сценариев использования.

- Декораторы  **MessagePattern** и **EventPattern** позволяют задать пользовательские параметры, но использовать их не возможно. Через декоратор **EventRabbitMqMessage** такие параметры будут доступны в `IConsumerDeserializer` и `RabbitMqContext.getMessageOptions()`.
